Introduction

Chapter Overview

This chapter presents the comprehensive research methodology employed in the design and implementation of a real-time web-based resource-sharing and referral platform specifically tailored for primary healthcare centers in Ibadan North-West Local Government Area, Oyo State, Nigeria.

Importance of System Analysis and Design

  • User Requirements Understanding: Ensures system meets actual PHC operational needs
  • Technical Robustness: Structured approach guarantees security and performance standards
  • Stakeholder Engagement: Continuous feedback integration throughout development
  • Scalability Planning: Architecture designed for future expansion across more PHCs
  • Risk Mitigation: Early identification of technical and operational constraints
Research Approach: Mixed-methods methodology combining Design Science Research with User-Centered Design principles, emphasizing practical problem-solving for healthcare environments.

System Design - System Architecture

Service-Oriented Architecture with Phoenix Framework

LAYERED SYSTEM ARCHITECTURE
Presentation Layer: React.js Frontend (Responsive Design)
↕ HTTPS/WebSocket Communication ↕
API Gateway: Phoenix Router (Rate Limiting, Authentication)
↕ Internal Service Calls ↕
Business Logic: Phoenix Contexts (Domain Services)
↕ Ecto Database Access ↕
Data Layer: PostgreSQL (ACID Compliance, Encryption)

Frontend Technology

  • React.js: Component-based UI
  • Tailwind CSS: Utility-first styling
  • React Query: Efficient API data fetching
  • Zustand: Lightweight state management

Backend Technology

  • Elixir/Phoenix: Real-time, fault-tolerant
  • Phoenix LiveView: Real-time UI updates
  • Ecto: Database wrapper and query builder
  • Guardian: JWT authentication

Database & Infrastructure

  • PostgreSQL: Primary database
  • Redis: Session and cache management
  • Cloud Hosting: Scalable deployment
  • SSL/TLS: End-to-end encryption
Architecture Benefits: Actor model concurrency for handling thousands of concurrent PHC users, built-in fault tolerance, and real-time communication capabilities essential for healthcare coordination.

Core Entities

  • USER: Healthcare staff (id, name, email, role, facility_id)
  • FACILITY: PHCs (id, name, address, coordinates, lga)
  • RESOURCE: Medical supplies (id, name, category, unit)
  • INVENTORY: Stock levels (id, resource_id, facility_id, quantity, expiry)
  • RESOURCE_REQUEST: Sharing requests (id, requester_id, resource_id, status)

Key Relationships

  • 1:M - User belongs to one Facility
  • 1:M - Facility has many Inventory items
  • M:1 - Requests relate to one Resource
  • 1:M - User creates many Requests
  • M:M - Resources tracked across Facilities
